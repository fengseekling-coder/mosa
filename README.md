# MOSA

> **Every visual, in context.** 让每一张图，都有来处，也有下一次。

MOSA is a local-first creative asset library for Codex, Cowart, and the local Grok Build CLI. It automatically collects generated images and videos with their Prompt, generation context, provenance, and version history, then makes them searchable and reusable in one local Web app.

MOSA does not generate media or replace Codex/Cowart/Grok. Those tools create; MOSA preserves the assets and the context needed to reuse them.

中文使用说明见 [docs/guide.zh-CN.md](docs/guide.zh-CN.md)。

## What MOSA Does

- Automatically archives images generated by Codex, including the available task Prompt, model, task ID, source path, image hash, dimensions, and timestamp.
- Automatically archives images and videos generated by the local Grok Build CLI under `~/.grok/sessions`, including tool prompt, session ID, model/tool name, source path, hash, and media kind.
- Watches the dedicated MOSA Cowart canvas and safely discovers project canvases opened through real local Codex/Cowart launch events.
- Stores migrated runtime data in a local SQLite library with FTS5 search, cursor pagination, original images, WebP previews, thumbnails, tags, and provenance.
- Keeps recipe history as an asset-based version tree. A version can preserve the parent image as a snapshot or save a newly generated child image.
- Lets users inspect, edit, search, archive, duplicate, and insert a library image back into an approved Cowart canvas.

## Scope and Privacy

- The product is a local Web UI served on `127.0.0.1`; it is not a cloud service and must not be exposed through a reverse proxy or public port.
- Codex collection reads only the configured generated-image root and matching local Codex session records. It does not scan Downloads, Desktop, or arbitrary image folders.
- Grok collection reads only the configured Grok sessions root (`GROK_SESSIONS_DIR`, default `~/.grok/sessions`) and matching session chat history under that root. It does not call the Grok API or scan arbitrary folders.
- Cowart collection listens only to the dedicated MOSA canvas and registered or safely auto-discovered project canvases. It does not scan arbitrary projects.
- MOSA does not add an AI model, remote synchronization, embedding search, or a desktop `.app` in the current release.

## Requirements

- macOS with Node.js 22 or newer and npm.
- Codex Desktop for automatic Codex image collection.
- The Cowart plugin is optional and required only for Cowart collection or one-click canvas insertion.

## Quick Start

This starts a developer/local service on the default port, `43517`.

```bash
git clone https://github.com/fengseekling-coder/mosa.git
cd mosa
npm ci
npm test
npm start
```

Open <http://127.0.0.1:43517>.

The repository contains application code only. MOSA keeps each user's library, generated media, Prompt records, and provenance outside the checkout. For migration, health checks, or recovery, read [docs/operations.md](docs/operations.md).

## Local Library and Migration

MOSA preserves the legacy JSON store until a verified migration succeeds. The default independent runtime library is `~/MOSA Library` and contains:

```text
~/MOSA Library/
├── mosa.db                # SQLite runtime authority after migration
├── assets/<project>/
│   ├── original/          # Original image bytes, never modified
│   ├── previews/          # WebP, longest edge 1600px
│   └── thumbnails/        # WebP, longest edge 400px
└── legacy-json-backup/    # JSON and Prompt backup made during migration
```

```bash
# Inspect a legacy library without writing anything.
npm exec mosa -- migrate --dry-run --library /absolute/path/to/library

# Migrate, verify, and rebuild resumable derivative jobs.
npm exec mosa -- migrate --library /absolute/path/to/library
npm exec mosa -- verify --library /absolute/path/to/library
npm exec mosa -- thumbnails rebuild --library /absolute/path/to/library
```

Migration checks JSON, collection data, original-image hashes, and empty groups. A failed or incomplete migration does not activate SQLite. After a completed migration, SQLite is the only runtime metadata authority; JSON remains a backup and fallback path, not a dual-written store.

Set `MOSA_LIBRARY_DIR` only when selecting a non-default library:

```bash
MOSA_LIBRARY_DIR=/absolute/path/to/library npm start
```

## Automatic Collection

### Codex images

While the MOSA service is running, it watches `~/.codex/generated_images/`. For a matched task, MOSA reads the corresponding local session JSONL record and prefers the `image_generation_end.revised_prompt` actually used for image generation. If that event is unavailable, it records the task's latest user instruction and marks the Prompt status accordingly. If no matching session record is available, the asset is archived with an explicit unavailable status rather than a fabricated Prompt.

The original image is hard-linked when possible and copied only when the source and library are on different filesystems.

### Grok Build CLI media

While the MOSA service is running, it watches `~/.grok/sessions/` (override with `GROK_SESSIONS_DIR`). It discovers media only under session `images/` and `videos/` folders, then reads the matching session `chat_history.jsonl` for tool provenance.

Prompt priority:

1. The generation tool argument prompt when a tool call and tool result both match that media path (`prompt_status: generation-tool-prompt`).
2. The session user query only for that same matched tool result when the tool omitted a prompt (`prompt_status: session-user-prompt`).
3. An explicit unavailable status for orphan on-disk media or unmatched results (`prompt_status: not-available`).

Videos are archived and served as original media. MOSA does not run sharp, ffmpeg, or any transcoding on video files. The UI uses native video playback and exposes an **Open original media** action when browser playback is unavailable or insufficient.

### Cowart canvases

MOSA always monitors its dedicated canvas at `~/.codex/cowart-data/mosa/`. It can also discover a project canvas only from a real local `render_cowart_canvas_widget` or Cowart launcher event in a Codex session, verifies the project canvas marker, and then persists that approved canvas in the local registry.

Cowart snapshots provide canvas description and provenance, not a complete image-generation Prompt. MOSA marks this distinction so it is visible in asset metadata.

## Reuse and Version History

- The gallery loads thumbnails; the inspector uses previews; the original URL remains compatible at `/library/<project>/images/<file>`.
- **Save recipe** updates the current asset without creating a version.
- **Save as new version** requires `version_change`. Supplying `imagePath` saves a newly generated image as a child; omitting it copies the parent image as a recipe snapshot.
- `asset_duplicate` creates a separate version root and records `duplicated_from` provenance.
- Version history is a stable depth-first tree rooted at the original asset, including archived nodes.
- The asset inspector can insert an asset into a selected, approved Cowart canvas. Inserted assets keep a MOSA source ID so the bridge does not archive the same image again.

## Health Checks

Use the combined status endpoint for current bridge and Cowart discovery state:

```bash
curl -sS http://127.0.0.1:43517/api/bridges
curl -sS http://127.0.0.1:43517/api/cowart-canvases
curl -sS http://127.0.0.1:43517/api/library-path
```

`/api/bridges` reports Codex, Grok, and Cowart watcher/polling state, errors, discovered canvases, and Cowart insertion availability. A running service is required for new media to be collected automatically.

## HTTP and MCP Interfaces

MOSA keeps its existing asset fields while adding cursor pagination to `GET /api/assets` and MCP `asset_list`:

- `limit`: defaults to `100`, maximum `250`.
- `cursor`: requests the next stable page.
- response page data includes `page.total` and `page.nextCursor`.

Recipe-version routes:

```text
GET  /api/assets/:project/:asset/versions
POST /api/assets/:project/:asset/versions
```

The local MCP server exposes:

```text
asset_create
asset_list
asset_get
asset_update_metadata
asset_attach_prompt
asset_archive
asset_duplicate
asset_version_create
asset_version_history
```

Register it from the workspace parent directory, using absolute paths for your checkout:

```bash
codex mcp add mosa \
  --env MOSA_PROJECT_DIR=/absolute/path/to/your/workspace \
  -- node /absolute/path/to/mosa/mcp/server.mjs
```

For new child versions, pass the image tool's real `imagePath` to `asset_version_create` and include a non-empty `version_change` summary.

## Configuration

| Environment variable | Default | Purpose |
| --- | --- | --- |
| `MOSA_PROJECT_DIR` | Parent of the MOSA checkout | Workspace context for the service |
| `MOSA_PORT` | `43517` | Local HTTP port |
| `MOSA_LIBRARY_DIR` | `~/MOSA Library` | Runtime SQLite library after migration |
| `CODEX_GENERATED_IMAGES_DIR` | `~/.codex/generated_images` | Allowed Codex image source root |
| `CODEX_SESSIONS_DIR` | `~/.codex/sessions` | Local Codex session records used for Prompt provenance |
| `GROK_SESSIONS_DIR` | `~/.grok/sessions` | Local Grok Build CLI sessions root for media auto-archive |
| `COWART_MOSA_CANVAS_DIR` | `~/.codex/cowart-data/mosa` | Dedicated MOSA Cowart canvas |
| `MOSA_COWART_REGISTRY_PATH` | `~/.codex/mosa/cowart-projects.json` | Persisted approved project-canvas registry |
| `COWART_MCP_SERVER_PATH` | Cowart plugin server path | Optional MCP entry used for one-click Cowart insertion |
